# devmem — SOTA Developer Memory System

**Date:** 2026-03-23
**Author:** Arbaz
**Status:** Draft
**Stack:** Go + SQLite (single binary, zero external dependencies)

---

## 1. Problem Statement

Every AI coding CLI (Claude Code, Codex, Cursor, Windsurf, Gemini CLI) suffers from amnesia. When you close a session, all context is lost. Developers waste significant time re-explaining:

- What project they're working on
- What feature/task they were building
- What decisions were made and why
- What the plan was and how far they got
- What was tried and failed

Existing solutions (Mem0, Zep, Supermemory, Letta, KeepGoing, SaveContext, Engram, mcp-memory-keeper) each solve fragments of this problem but none combine:

1. Feature-scoped session tracking with git commit association
2. Bi-temporal fact storage with contradiction resolution
3. Semantic code diff understanding (entity-level, not line-level)
4. Plan persistence with auto-progress tracking
5. Background memory consolidation
6. Cross-tool compatibility via MCP standard
7. Single binary, zero dependencies, local-first

**devmem** fills this gap.

## 2. Core Concepts

### 2.1 Feature (top-level entity)

A feature represents a unit of work: "auth-v2-migration", "billing-fix", "onboarding-flow". Features group sessions, commits, notes, plans, and facts together.

- Created explicitly by the developer ("let's work on auth-v2")
- Auto-linked to a git branch when possible
- Has a lifecycle: `active` → `paused` → `done`
- Multiple features can exist per project; only one is active at a time

### 2.2 Session (automatic, per CLI connection)

Every time an MCP client connects, a new session is created under the active feature. Sessions are never manually managed.

- Tracks which tool connected (claude-code, cursor, codex, windsurf, unknown)
- Records start/end time
- All memories created during the session are linked to it

### 2.3 Bi-Temporal Facts

Every fact has two time axes (inspired by Zep/Graphiti, implemented in pure SQLite):

- `valid_at` — when this became true in the real world
- `invalid_at` — when this was superseded (NULL = still true)
- `recorded_at` — when the system learned this

Facts are never deleted, only invalidated. This enables:
- "What's true now?" — `WHERE invalid_at IS NULL`
- "What was true last week?" — `WHERE valid_at <= date AND (invalid_at IS NULL OR invalid_at > date)`
- "What changed between sessions?" — `WHERE recorded_at BETWEEN a AND b`

### 2.4 Memory Links (A-MEM / Zettelkasten)

Every memory (fact, note, commit, plan) can link to other memories. Links are bidirectional with a relationship type and strength score.

On insert, the system finds related memories via FTS5 and creates links automatically. When new information contradicts an existing memory, the old memory's context can be updated (memory evolution).

### 2.5 Plans (first-class entity)

Plans generated by any CLI are captured and persist across sessions. Plans have steps with individual status tracking. Commits are auto-matched to plan steps via FTS5 similarity.

Plans are bi-temporal: when requirements change and a new plan is generated, the old plan is invalidated (not deleted), preserving history.

### 2.6 Semantic Code Changes

Instead of storing "lines 45-67 changed", devmem uses tree-sitter to understand what code *entities* changed: "function ValidateToken was modified — added rate limiting check."

This is 2.3x more useful to AI agents than raw diffs (per research from Ataraxy-Labs' `sem` tool).

### 2.7 Recursive Summarization

When memories accumulate, they're compressed into tiered summaries:
- Generation 0: summarizes ~20 memories
- Generation 1: summarizes ~5 gen-0 summaries
- Generation N: logarithmic compression

10,000 memories compress to ~3 levels of summaries. The latest summary is always included in context retrieval.

### 2.8 Consolidation Engine

A background goroutine that runs on entropy triggers (not fixed schedules):
- Detects contradictions between facts
- Decays irrelevant memories (14-day half-life)
- Synthesizes patterns ("you've been refactoring auth all week")
- Discovers new links between memories
- Triggers recursive summarization when thresholds are met

### 3.1 Transport: stdio

The MCP server uses stdio transport (not SSE/HTTP). The MCP client spawns `devmem` as a subprocess and communicates via stdin/stdout. This means:
- No ports to configure
- No daemon to manage
- No Docker required
- Process lifecycle managed by the MCP client

### 3.2 Project Detection

On startup, devmem:
1. Gets the current working directory from the MCP client
2. Walks up to find `.git` root
3. Uses `<git-root>/.memory/` as the storage directory
4. Creates `.memory/` and adds to `.gitignore` if not present

### 3.3 Concurrent Access

Multiple MCP clients can connect to the same project simultaneously (e.g., Claude Code in terminal + Cursor in IDE). SQLite WAL mode handles this:
- Single writer connection (`SetMaxOpenConns(1)`)
- Multiple reader connections
- `busy_timeout=5000` for write contention
- `BEGIN IMMEDIATE` for write transactions

## 4. Data Model

### 4.1 Schema

```sql
-- ============================================================
-- FEATURES
-- ============================================================
CREATE TABLE features (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL UNIQUE,
    description TEXT DEFAULT '',
    status      TEXT NOT NULL DEFAULT 'active'
                CHECK (status IN ('active', 'paused', 'done')),
    branch      TEXT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    last_active TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_features_status ON features(status);

-- ============================================================
-- SESSIONS
-- ============================================================
CREATE TABLE sessions (
    id          TEXT PRIMARY KEY,
    feature_id  TEXT NOT NULL REFERENCES features(id),
    tool        TEXT NOT NULL DEFAULT 'unknown',
    started_at  TEXT NOT NULL DEFAULT (datetime('now')),
    ended_at    TEXT
);
CREATE INDEX idx_sessions_feature ON sessions(feature_id);

-- ============================================================
-- BI-TEMPORAL FACTS
-- ============================================================
CREATE TABLE facts (
    id          TEXT PRIMARY KEY,
    feature_id  TEXT NOT NULL REFERENCES features(id),
    session_id  TEXT REFERENCES sessions(id),
    subject     TEXT NOT NULL,
    predicate   TEXT NOT NULL,
    object      TEXT NOT NULL,
    valid_at    TEXT NOT NULL DEFAULT (datetime('now')),
    invalid_at  TEXT,
    recorded_at TEXT NOT NULL DEFAULT (datetime('now')),
    confidence  REAL NOT NULL DEFAULT 1.0
);
CREATE INDEX idx_facts_temporal ON facts(subject, predicate, invalid_at);
CREATE INDEX idx_facts_feature ON facts(feature_id);
CREATE INDEX idx_facts_valid ON facts(valid_at);

-- ============================================================
-- NOTES (progress, decisions, blockers, next_steps)
-- ============================================================
CREATE TABLE notes (
    id          TEXT PRIMARY KEY,
    feature_id  TEXT NOT NULL REFERENCES features(id),
    session_id  TEXT REFERENCES sessions(id),
    content     TEXT NOT NULL,
    type        TEXT NOT NULL DEFAULT 'note'
                CHECK (type IN ('progress', 'decision', 'blocker', 'next_step', 'note')),
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_notes_feature ON notes(feature_id);
CREATE INDEX idx_notes_type ON notes(type);

-- ============================================================
-- PLANS
-- ============================================================
CREATE TABLE plans (
    id          TEXT PRIMARY KEY,
    feature_id  TEXT NOT NULL REFERENCES features(id),
    session_id  TEXT REFERENCES sessions(id),
    title       TEXT NOT NULL,
    content     TEXT NOT NULL,
    status      TEXT NOT NULL DEFAULT 'active'
                CHECK (status IN ('active', 'completed', 'abandoned', 'superseded')),
    source_tool TEXT DEFAULT 'unknown',
    valid_at    TEXT NOT NULL DEFAULT (datetime('now')),
    invalid_at  TEXT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_plans_feature ON plans(feature_id);
CREATE INDEX idx_plans_status ON plans(status);

CREATE TABLE plan_steps (
    id              TEXT PRIMARY KEY,
    plan_id         TEXT NOT NULL REFERENCES plans(id) ON DELETE CASCADE,
    step_number     INTEGER NOT NULL,
    title           TEXT NOT NULL,
    description     TEXT DEFAULT '',
    status          TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'in_progress', 'completed', 'skipped')),
    completed_at    TEXT,
    linked_commits  TEXT DEFAULT '[]'
);
CREATE INDEX idx_plan_steps_plan ON plan_steps(plan_id);

-- ============================================================
-- GIT COMMITS
-- ============================================================
CREATE TABLE commits (
    id                  TEXT PRIMARY KEY,
    feature_id          TEXT NOT NULL REFERENCES features(id),
    session_id          TEXT REFERENCES sessions(id),
    hash                TEXT NOT NULL UNIQUE,
    message             TEXT NOT NULL,
    author              TEXT NOT NULL,
    files_changed       TEXT DEFAULT '[]',
    intent_type         TEXT DEFAULT 'unknown'
                        CHECK (intent_type IN (
                            'feature', 'bugfix', 'refactor', 'test',
                            'docs', 'infra', 'cleanup', 'unknown'
                        )),
    intent_confidence   REAL DEFAULT 0.0,
    committed_at        TEXT NOT NULL,
    synced_at           TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_commits_feature ON commits(feature_id);
CREATE INDEX idx_commits_hash ON commits(hash);

-- ============================================================
-- SEMANTIC CODE CHANGES (tree-sitter powered)
-- ============================================================
CREATE TABLE semantic_changes (
    id          TEXT PRIMARY KEY,
    commit_hash TEXT NOT NULL REFERENCES commits(hash),
    file_path   TEXT NOT NULL,
    entity_name TEXT NOT NULL,
    entity_kind TEXT NOT NULL
                CHECK (entity_kind IN (
                    'function', 'method', 'struct', 'class',
                    'interface', 'type', 'constant', 'variable',
                    'module', 'package', 'other'
                )),
    change_type TEXT NOT NULL
                CHECK (change_type IN ('added', 'modified', 'deleted', 'renamed')),
    old_name    TEXT,
    description TEXT DEFAULT '',
    session_id  TEXT REFERENCES sessions(id),
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_semantic_commit ON semantic_changes(commit_hash);
CREATE INDEX idx_semantic_entity ON semantic_changes(entity_name);

-- ============================================================
-- MEMORY LINKS (A-MEM / Zettelkasten connections)
-- ============================================================
CREATE TABLE memory_links (
    id              TEXT PRIMARY KEY,
    source_id       TEXT NOT NULL,
    source_type     TEXT NOT NULL
                    CHECK (source_type IN ('fact', 'note', 'commit', 'plan', 'plan_step', 'semantic_change')),
    target_id       TEXT NOT NULL,
    target_type     TEXT NOT NULL
                    CHECK (target_type IN ('fact', 'note', 'commit', 'plan', 'plan_step', 'semantic_change')),
    relationship    TEXT NOT NULL
                    CHECK (relationship IN (
                        'related', 'contradicts', 'extends', 'caused_by',
                        'implements', 'supersedes', 'blocks', 'resolves'
                    )),
    strength        REAL NOT NULL DEFAULT 0.5,
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(source_id, source_type, target_id, target_type, relationship)
);
CREATE INDEX idx_links_source ON memory_links(source_id, source_type);
CREATE INDEX idx_links_target ON memory_links(target_id, target_type);

-- ============================================================
-- RECURSIVE SUMMARIES
-- ============================================================
CREATE TABLE summaries (
    id          TEXT PRIMARY KEY,
    scope       TEXT NOT NULL,
    content     TEXT NOT NULL,
    generation  INTEGER NOT NULL DEFAULT 0,
    token_count INTEGER NOT NULL DEFAULT 0,
    covers_from TEXT NOT NULL,
    covers_to   TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_summaries_scope ON summaries(scope, generation);

-- ============================================================
-- CONSOLIDATION STATE
-- ============================================================
CREATE TABLE consolidation_state (
    id                  INTEGER PRIMARY KEY CHECK (id = 1),
    last_run_at         TEXT,
    entropy_score       REAL NOT NULL DEFAULT 0.0,
    unsummarized_count  INTEGER NOT NULL DEFAULT 0,
    conflict_count      INTEGER NOT NULL DEFAULT 0,
    next_trigger_at     TEXT
);
INSERT INTO consolidation_state (id) VALUES (1);

-- ============================================================
-- FTS5 VIRTUAL TABLES
-- ============================================================
CREATE VIRTUAL TABLE notes_fts USING fts5(
    content, type,
    content='notes',
    content_rowid='rowid',
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE commits_fts USING fts5(
    message,
    content='commits',
    content_rowid='rowid',
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE facts_fts USING fts5(
    subject, predicate, object,
    content='facts',
    content_rowid='rowid',
    tokenize='porter unicode61'
);

CREATE VIRTUAL TABLE plans_fts USING fts5(
    title, content,
    content='plans',
    content_rowid='rowid',
    tokenize='porter unicode61'
);

-- FTS5 trigram tables for substring matching

    content='notes',

);

    content='commits',
    content_rowid='rowid',

);
```

#### `devmem_search`

3-layer search across all memory types.

```
Parameters:
    query: string (required)
    scope: string? ("current_feature" | "all_features", default "current_feature")
    types: string[]? (["notes", "facts", "commits", "plans", "semantic_changes"], default all)
    as_of: datetime? (bi-temporal — search state at a point in time)
Returns: [{
    type: string,
    content: string,
    relevance: float,
    feature: string,
    created_at: datetime,
    links: [{ content, relationship }]
}]
```

**3-layer search (inspired by Context Mode):**
1. FTS5 + BM25 with porter stemming
2. Trigram substring matching (catches partial matches)
3. Fuzzy correction via Levenshtein distance on vocabulary table

Results are boosted by:
- Type weight (decision=2x, blocker=1.5x, progress=1x, note=0.5x)
- Temporal decay (14-day half-life)
- Link connectivity (more links = higher boost)

## 6. Search Engine

### 6.1 Three-Layer Fallback

```
Query: "token refresh"
         │
         ▼
Layer 1: FTS5 + BM25 (porter stemming)
         "token" stems to "token", "refresh" matches "refresh"
         Results ranked by BM25(notes_fts, 2.0, 1.0)
         │
         ├── Found results? → return (sorted by relevance * decay * type_weight)
         │
         ▼ (no results)
Layer 2: Trigram substring
         "tok" matches "tokenization", "ref" matches "refreshing"
         │
         ├── Found results? → return
         │
         ▼ (no results)
Layer 3: Fuzzy correction
         Levenshtein distance against vocabulary table
         "tokn" corrects to "token" (distance=1)
         Re-run Layer 1 + 2 with corrected terms
```

### 6.2 Scoring Formula

```
score = bm25_score * temporal_decay * type_weight * link_boost

temporal_decay = exp(-0.693 * days_since_creation / 14)  // 14-day half-life
type_weight    = { decision: 2.0, blocker: 1.5, progress: 1.0, note: 0.5 }
link_boost     = 1.0 + (link_count * 0.1)  // more connected = more relevant
```

### 6.3 Graph Traversal

After FTS5 returns results, traverse `memory_links` using recursive CTEs to find connected memories within 2 hops:

```sql
WITH RECURSIVE connected AS (
    SELECT target_id, target_type, relationship, strength, 1 as depth
    FROM memory_links
    WHERE source_id = ? AND source_type = ?
    UNION ALL
    SELECT ml.target_id, ml.target_type, ml.relationship, ml.strength, c.depth + 1
    FROM memory_links ml
    JOIN connected c ON ml.source_id = c.target_id AND ml.source_type = c.target_type
    WHERE c.depth < 2
)
SELECT DISTINCT * FROM connected ORDER BY depth, strength DESC;
```

## 7. Git Engine

### 7.1 Commit Ingestion

```
git log --since=<since> --format='%H|%s|%an|%ai' --no-merges
```

For each commit:
1. Parse hash, message, author, date
2. Get files: `git diff-tree --no-commit-id --name-status -r <hash>`
3. Classify intent
4. Run semantic diff

### 7.2 Intent Classification (Heuristic)

No ML model needed. Keyword + file path signals:

```go
func classifyIntent(message string, files []string) (string, float64) {
    message = strings.ToLower(message)

    // Message keywords (highest priority)
    if matchesAny(message, "fix", "bug", "patch", "resolve", "crash", "issue") {
        return "bugfix", 0.8
    }
    if matchesAny(message, "feat", "add", "implement", "support", "introduce") {
        return "feature", 0.8
    }
    if matchesAny(message, "refactor", "clean", "rename", "simplify", "reorganize") {
        return "refactor", 0.8
    }
    if matchesAny(message, "test", "spec", "coverage") {
        return "test", 0.8
    }
    if matchesAny(message, "doc", "readme", "comment") {
        return "docs", 0.8
    }

    // File path signals (lower priority)
    if allMatch(files, "*_test.go", "*.test.*", "*.spec.*") {
        return "test", 0.6
    }
    if allMatch(files, "Dockerfile", "*.yml", "*.yaml", ".github/*") {
        return "infra", 0.6
    }

    // Diff statistics
    // High deletion ratio → cleanup/refactor
    // New files only → feature

    return "unknown", 0.0
}
```

### 7.3 Semantic Diff (tree-sitter)

Using `github.com/tree-sitter/go-tree-sitter` with language grammars:

1. Get old and new file content: `git show <hash>~1:<path>` and `git show <hash>:<path>`
2. Parse both into ASTs
3. Extract top-level entities (functions, structs, methods, constants)
4. Three-phase matching:
   - Phase 1: Exact name match
   - Phase 2: Content hash match (renamed but identical body)
   - Phase 3: Fuzzy similarity (renamed AND modified)
5. Classify each entity: added/modified/deleted/renamed
6. Store in `semantic_changes` table

**Supported languages (initial):**
- Go (primary — your stack)
- TypeScript/JavaScript (your frontend stack)
- More added via tree-sitter grammars as needed

## 8. Consolidation Engine

### 8.1 Trigger Conditions

```go
type ConsolidationTrigger struct {
    EntropyThreshold    float64 // default 0.7
    IdleTimeout         time.Duration // default 5 minutes of no writes
    MaxUnsummarized     int // default 20
    MaxConflicts        int // default 3
}
```

Entropy score formula:
```
entropy = (unsummarized_count / 20) * 0.4 +
          (conflict_count / 3) * 0.3 +
          (time_since_last_run_hours / 24) * 0.3
```

When `entropy >= threshold`, consolidation runs.

### 8.2 Consolidation Steps

1. **Contradiction detection:**
   ```sql
   SELECT subject, predicate, COUNT(*) as cnt
   FROM facts WHERE invalid_at IS NULL
   GROUP BY subject, predicate HAVING cnt > 1
   ```
   For each conflict: keep the most recent (by valid_at), invalidate others.

2. **Memory decay:**
   ```sql
   UPDATE notes SET relevance_score = relevance_score * 0.95
   WHERE updated_at < datetime('now', '-7 days')
   AND last_accessed < datetime('now', '-7 days')
   ```

3. **Link discovery:**
   For each unlinked note created since last consolidation:
   - Run FTS5 search with note content as query
   - For each match with BM25 score > threshold → create link

4. **Pattern synthesis:**
   Group recent notes/commits by feature and tag clusters.
   If a cluster forms (5+ related items in the last week) → generate synthesis note.

5. **Recursive summarization:**
   If unsummarized count > threshold → create gen-0 summary.
   If gen-0 count > 5 → create gen-1 summary.

### 8.3 No LLM Required for V1

The consolidation engine in V1 uses only SQL queries and heuristics. No LLM calls. This keeps the binary self-contained with zero external dependencies.

Future versions can optionally use an LLM for:
- Better contradiction resolution
- Richer pattern synthesis
- More nuanced summarization

## 9. Plan Engine

### 9.1 Plan Auto-Detection

When `devmem_remember` receives content, check for plan-like structure:

```go
func isPlanLike(content string) bool {
    lines := strings.Split(content, "\n")
    numberedLines := 0
    for _, line := range lines {
        trimmed := strings.TrimSpace(line)
        if regexp.MustCompile(`^\d+[\.\)]\s+`).MatchString(trimmed) {
            numberedLines++
        }
    }
    // Plan-like if 3+ numbered items and contains plan-related keywords
    hasPlanKeyword := containsAny(content, "plan", "steps", "todo", "phase", "milestone")
    return numberedLines >= 3 && hasPlanKeyword
}
```

If detected → parse steps → create plan + plan_steps → return `promoted_to_plan: true`.

### 9.2 Commit-to-Step Matching

After syncing commits, try to match each commit to an incomplete plan step:

```go
func matchCommitToStep(commitMessage string, steps []PlanStep) *PlanStep {
    for _, step := range steps {
        if step.Status == "completed" || step.Status == "skipped" {
            continue
        }
        // FTS5 similarity between commit message and step title+description
        score := fts5Similarity(commitMessage, step.Title + " " + step.Description)
        if score > MATCH_THRESHOLD {
            return &step
        }
    }
    return nil
}
```

### 9.3 Plan Versioning (Bi-Temporal)

When a new plan supersedes an old one:
1. Old plan: `invalid_at = now()`, `status = 'superseded'`
2. New plan: `valid_at = now()`
3. Copy completed steps from old plan to new plan (preserving commit links)
4. Create link: new_plan `supersedes` old_plan

## 11. Dependencies

### Go Modules

```
github.com/mark3labs/mcp-go          # MCP server SDK
github.com/mattn/go-sqlite3          # SQLite driver (CGO)
  OR modernc.org/sqlite              # Pure Go SQLite (no CGO)
github.com/tree-sitter/go-tree-sitter  # AST parsing
github.com/google/uuid               # UUID generation
```

**Decision: CGO vs Pure Go SQLite**

- `mattn/go-sqlite3` (CGO): Faster, supports FTS5 out of the box, but requires C compiler
- `modernc.org/sqlite` (Pure Go): No CGO needed, single binary cross-compile, FTS5 support

**Recommendation:** Start with `modernc.org/sqlite` for true zero-dependency single binary. Switch to `mattn/go-sqlite3` only if FTS5 performance becomes a bottleneck.

### tree-sitter Grammars

- `github.com/tree-sitter/tree-sitter-go`
- `github.com/tree-sitter/tree-sitter-javascript`
- `github.com/tree-sitter/tree-sitter-typescript`

### External Tools (not Go deps, used at runtime)

- `git` — expected to be installed on the machine (reasonable assumption)
